# Building language bindings for FKL

Proven architecture: the Python front-end (fkl-python, built on this
design) runs 12 test suites x 2 compilers with ~19-70 us/call overhead
and zero recompiles on value changes.

## Why the obvious approaches fail

1. **pybind11/SWIG over a fixed API**: impossible — there is no fixed
   API. The kernel is GENERATED by the C++ type system from the user's
   op sequence; every distinct chain is a new template instantiation.
2. **NVRTC**: dead end. NVRTC compiles DEVICE code only, but FKL's fusion
   machinery (`build()`, `BackFuser`, `executeOperations`, grid
   computation) is HOST code. You cannot fuse with NVRTC alone.
3. **Pre-compiled "bag of kernels"**: defeats the whole point — the
   combinatorial space of chains is unbounded.

## The architecture that works

```
user composes lazily (no compile)
        |
        v   first call with concrete dtype/shape
generate ONE .cu translation unit:
    extern "C" void fkl_entry(void* in, void* out, const Dims*,
                              const float* params, void* stream)
    { ...build IOps... executeOperations<TransformDPP<>>(...); }
        |
        v
single-step host+device compile to .so   (clang preferred, nvcc fallback)
        |
        v
disk-cache keyed by TYPE SIGNATURE; dlopen + one FFI call per launch
```

Key decisions:

1. **Types compile, values don't.** The cache key contains: op tokens
   (names + template args), dtypes, layout (2D vs planes), batch size,
   GPU arch, compiler, FKL/CUDA versions, and a codegen-version counter.
   Runtime values (factors, rects, matrices, sizes) are passed through a
   flat `float params[]` at every call. Changing a value NEVER recompiles;
   changing the chain shape compiles exactly once.
2. **Generate what a C++ user would write.** The binding emits the same
   `executeOperations` call from the README — never reimplement fusion
   logic in the host language.
3. **clang first.** clang compiles host+device CUDA in one invocation and
   is Apache-2.0 (no NVIDIA EULA in your wheel/package); fall back to
   nvcc when clang is missing or too old. Newer CUDA may need shims for
   clang (texture header stubs, fatbinary flag translation) — keep them
   in the binding, auto-applied.
4. **Bump a CODEGEN_VERSION constant in the cache key whenever the
   emitted C++ changes**, or stale cached .so files will silently serve
   old code.

## The ABI

Keep `fkl_entry` dumb and stable:
- `void* d_in, void* d_out`: raw device pointers (or `void**` array for
  image batches).
- `Dims*`: width/height/planes for in and out — lets ONE compiled kernel
  serve any image size with the same signature.
- `const float* params`: all runtime values, in chain order. The binding
  computes each op's slice offset at generation time.
- `void* stream`: external CUDA stream. If null, use a TU-local
  `static Stream` (creating/destroying a stream per call costs ~15 us)
  and sync before returning; if provided, run async and DON'T sync —
  the caller owns ordering (torch/cupy semantics).

## Zero-copy interop

- Inputs: accept anything exposing a device pointer + shape + dtype
  (`__cuda_array_interface__` in Python). Wrap with
  `Ptr2D<T>(ptr, w, h, pitch, MemType::Device)` — no copy.
  Require C-contiguous or honor strides via the pitch argument.
- Outputs: allocate with the CUDA driver API and expose BOTH the array
  interface and DLPack so frameworks adopt the memory without copying.
  On DLPack export, transfer ownership to the consumer's deleter and
  disarm your own destructor (double-free guard).

## Mapping the fusion techniques

| technique | binding surface |
|---|---|
| VF | ordered list of op descriptors between read and write |
| BVF | geometry descriptors right after the read; emit ReadBack builds |
| HF | LISTS of parameters: list of rects -> `std::array<Rect,N>`; list of images -> `std::array<Ptr2D<T>,N>` (N is part of the type => one compile per N) |
| DHF | list of chains + plane->sequence map; generate a SequenceSelector struct |
| CircularTensor | stateful handle (create/update/snapshot/destroy in the same .so) |

## Performance budget (measured reference points)

- steady-state fused launch: ~19-70 us/call (one FFI call, no sync path)
- cold compile: 1-3 s per signature, once, then disk-cached forever
- fused 6-op chain at 1080p: ~5x faster than 6 separate kernel launches

## Testing a binding

- Validate against CPU references computed in the host language, not
  against FKL itself.
- Always test BOTH compilers (clang + nvcc), wiping the cache between.
- Test that value changes hit the cache (same .so path) and that chain
  changes miss it.
- Position-encode test images (value = f(x, y, frame)) so wrong reads
  reveal WHERE they read.
